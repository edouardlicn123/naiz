# 注入工具故障排查与修复记录

## 背景

`makegame.sh make demo-A1` 制作的启动盘在 NP2kai 模拟器上出现启动故障。

**预期行为**：MS-DOS 5.0 从 SCSI HDD (C:) 启动 → 处理 CONFIG.SYS → 进入 COMMAND.COM → 执行 AUTOEXEC.BAT → 自动运行 ENGINE.EXE。

**实际行为**：启动后出现"询问文件数"提示，或 "找不到C盘" 报错，无法进入游戏。

---

## 通过二分法定位的问题

### 问题 1："询问文件数"提示

**现象**：MS-DOS 启动过程中（内存检测后、CONFIG.SYS 处理前/期间）弹出提示，询问用户有多少个文件。

**排查过程**：
1. 测试 `test-autoexec.hdi`（仅替换 AUTOEXEC.BAT，原版 CONFIG.SYS）→ 无此提示
2. 测试 `test-config.hdi`（仅替换 CONFIG.SYS，原版 AUTOEXEC.BAT）→ 无此提示
3. 测试 `test-nodbl.hdi`（完整注入，但去除 DBLSPACE.BIN）→ **无此提示**
4. 结论：DBLSPACE.BIN 驱动的磁盘扫描误判游戏文件数据为压缩卷（CVF），触发交互式提示

**修复**：
- 在 `inject_into_hdi()` 中新增 Step 0：自动从根目录移除 DBLSPACE.BIN（标记为 0xE5）
- 文件：`tools/naiz_img/inject_common.py`

### 问题 2："找不到C盘"报错

**现象**：AUTOEXEC.BAT 执行时报错，无法访问 C: 驱动器。

**排查过程**：
1. 测试 `test-config.hdi`（新 CONFIG.SYS 含 C:\ 路径，原版 AUTOEXEC.BAT）→ 正常进入 DOSSHELL
2. 测试 `test-aeonly.hdi`（原版 CONFIG.SYS + 错误重写的 AUTOEXEC.BAT）→ "找不到C盘"
3. 对比 `test-aeonly.hdi` 与基座镜像的字节差异 → 发现重写 AUTOEXEC.BAT 时写入了错误内容
4. 读取基座镜像的原始 AUTOEXEC.BAT（94 字节），发现内容**并非我之前猜测的 `C:\DOS\DOSSHELL.EXE`**：
   - 原版内容：
     ```
     @ECHO OFF
     PATH A:\DOS;A:\
     SET TEMP=A:\DOS
     SET DOSDIR=A:\DOS
     MOUSE
     DOSSHELL
     MOUSE /R
     ```
   - 我写的错误版本（85 字节）：用 `C:\DOS\DOSSHELL.EXE` 代替了 `MOUSE / DOSSHELL / MOUSE /R`
5. 测试 `test-exact94.hdi`（精确写入原版 94 字节，簇剩余部分清零）→ **正常进入 DOSSHELL**，证实问题在我的内容

**修复**：
- 重写 `generate_autoexec()`：移除 `C:\` 路径，改用 `\` 根相对路径；去除 `ECHO BOOT_OK > C:\BOOTMARK.TXT` 写文件操作；简化内容

改动前 `generate_autoexec` 输出（130 字节）：
```
@ECHO OFF
PATH C:\DOS;C:\DEMO-A1
SET TEMP=C:\DOS
SET DOSDIR=C:\DOS
ECHO BOOT_OK > C:\BOOTMARK.TXT
CD C:\DEMO-A1
ENGINE.EXE
```

改动后输出（79 字节）：
```
@ECHO OFF
PATH \DOS
SET TEMP=\DOS
SET DOSDIR=\DOS
CD \DEMO-A1
ENGINE.EXE
```

- 文件：`tools/naiz_img/inject_common.py` 第 22-32 行

### 问题 3：CONFIG.SYS 设备驱动路径错误

**现象**：CONFIG.SYS 中所有 `DEVICE=`, `DEVICEHIGH=` 行用了 `C:\` 前缀，但 MS-DOS 在处理 CONFIG.SYS 时盘符（C:）尚未分配。

**修复**：
- 将所有 `C:\` 改为 `\`（根相对路径），包括：
  - `DEVICE=C:\DOS\HIMEM.SYS` → `DEVICE=\DOS\HIMEM.SYS`
  - `DEVICE=C:\DOS\EMM386.EXE /P=448 /UMB /T=C:\DOS\EXTDSWAP.SYS` → `DEVICE=\DOS\EMM386.EXE /P=448 /UMB /T=\DOS\EXTDSWAP.SYS`
  - 以及 PRINT.SYS, SMARTDRV.SYS, RSDRV.SYS, KKCFUNC.SYS 等
  - NECAIK2.DRV 的参数 `C:NECAI.SYS` → `\NECAI.SYS`（NECAI.SYS 在根目录）
- 文件：`tools/ref_config/CONFIG.SYS`

### 问题 4：ENGINE.EXE 未自动运行（待修复）

**现象**：新制作的 demo-A1.hdi 启动后停在命令提示符，未执行 `CD \DEMO-A1` 和 `ENGINE.EXE`。

**根因**（已定位，未修复）：
- 注入时创建游戏目录（cluster 78）后**未清零簇数据**
- Cluster 78 之前被分配给其他 EXE 文件（原基座镜像中的分配历史），残留的 EXE 二进制数据（`MZ%...`）被注入代码的 `existing_entries` 解析逻辑误识别为合法的目录项
- 残留的垃圾目录项干扰了 MS-DOS 的目录解析，导致 `CD \DEMO-A1` 失败
- 间接原因：`_alloc()` 扫描空闲簇时没有越过 DBLSPACE.BIN 的旧 FAT 链（簇 81-112），导致游戏文件分配跳跃

**待修复方案**：
- 在 Step 3 创建目录后立即写入零：`_write_cluster(fs, dir_cluster, b'\x00' * fs.cluster_size)`
- 或者在 Step 4 读取 `dir_data` 前清零目录簇
