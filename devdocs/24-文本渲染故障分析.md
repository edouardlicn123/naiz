# 文本渲染故障分析

## 当前症状

NP2kai 模拟器中运行 demo-A1 时，文字框区域显示：
- 第一行：一堆空格或 tab 后一个句号
- 第二行：光标闪烁（文字淡入动画的视觉效果）

调试文本（main.c:64-66，白色/灰色/黄色，无动画）的可见性未知。

## 已知正常的部分

- 引擎启动正常，背景图像加载，文字框九宫格绘制正常
- MS-DOS 5.0 系统引导正常
- 文本数据格式解析正确：num_texts=48，压缩数据位置正确
- ASCII 字型数据已嵌入引擎（`font_ascii.c`，94 字符 × 16 字节）

## 尝试过的修复

### 1. preprocess_string escape sequence bug

**文件**: `core/engine/textengine.c:480-507`

**问题**: 第二遍扫描（自动换行）中，`case 0x1B` 设置 `uch = 0x0420` 后落入 copy 分支，导致每个转义序列复制 3 字节垃圾。

**修复**: 确认存在，但 demo-A1 第一条场景文本（"Well, would you look at that…"）不包含转义序列，因此不是主因。

### 2. swap_char_data_formats bit-reverse

**问题**: `swap_char_data_formats` 对字型数据做逐位反转，怀疑是 bug。

**结论**: 这是 planar↔packed 像素格式转换的必要操作（PC-98 每个位平面 16-bit → EGC 期望的 packed 16-bit×4）。**不是 bug。**

### 3. EGC 模式污染 — draw_char

**文件**: `core/engine/textengine.c:252-256`

**问题**: `draw_char()` 只设 `bit_length` 和 `bit_address`，不恢复 EGC mono draw 模式。背景/文本框绘制（`egc_set_to_vram_blit()`）后 EGC source=VRAM，导致字符数据写入时 EGC 从 VRAM 读取而非使用 CPU 源数据。

**修复**: 在 `draw_char()` 开头加入 `egc_set_to_mono_draw_mode()`。

**结果**: 未改善。因场景文字走动画路径使用 `draw_char_mask()`，而非 `draw_char()`。

### 4. EGC 模式污染 — draw_char_mask

**文件**: `core/engine/textengine.c:333-337`

**问题**: 同上，`draw_char_mask()` 也未恢复 EGC 模式。

**修复**: 在 `draw_char_mask()` 开头加入 `egc_set_to_mono_draw_mode()`。

**状态**: 已提交但用户尚未测试。

## 仍可能的问题

### A. 动画缓冲区初始状态

`string_write_animation_frame` 使用 16 slot 循环缓冲区。前 16 帧中大部分 slot 未填入有效字符数据（`anim_char_buf` 虽清零但 fade mask 从 0 递增）。前几帧的视觉输出 = 垃圾/空白 + 第一个可见字符。若 EGC 模式不对，全程空白。

### B. LZ4 解压结果

`lz48086.asm` 汇编解压器忽略 `dst_size` 参数（`cmp_data` 格式为 `[4-byte compressed_size][LZ4 data]`）。若 TXA 中 scene text 的 compressed_size 或数据偏移计算有误，解压结果将为空或垃圾。

验证方法：在 `load_scene_text` 中解压后、`preprocess_string` 中打印场景文本前 32 字节到 `ENGINE.LOG`。

### C. 场景字节码未调用文字渲染

检查 `SCENE.DAT` 是否包含 opcode 0x10（显示文字）的正确参数。若 `next_text_num` 或场景编号不对，可能渲染空字符串或无效偏移。

### D. 字型缓存失效

`init_font_file()` 在找不到 `FONT.DAT` 时静默返回（不报错）。`load_glyph_cache_with_char` 依赖 `ascii_char_cache`（`font_ascii.c`）。检查 Unicode 码点映射是否正确（ASCII 0x21-0x7E 对应 `font_ascii.c` 中索引 0-93）。

### E. debug 文本（main.c:64-66）是否可见

确认方法：在模拟器中观察屏幕左上角（y=16,32,48）。若可见，说明 `draw_char` 的 EGC 修复有效，问题在动画路径。若不可见，说明 EGC 修复无效或根本未进入渲染循环。

## 下一步建议

1. 确认 `load_scene_text` → `lz4_decompress` → `preprocess_string` 路径数据正确（加日志输出到 ENGINE.LOG）
2. 确认 `string_write_animation_frame` 正常执行并调用了 `draw_char_mask`
3. 缩小范围：禁用动画，直接调 `write_string_internal` 渲染场景文本
4. 检查 PC-98 VRAM 内容（直接读取 GDC_PLANES 段）确认是否有像素写入
