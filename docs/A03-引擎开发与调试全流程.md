## 2. 模拟器概要（NP2kai）

### 两大核心

| 核心 | 二进制 | 模拟目标 |
|------|--------|----------|
| **[DEPRECATED] i286** | `sdlnp2kai_sdl2` | PC-9801 (286) |
| **IA32**（当前使用） | `sdlnp21kai_sdl2` | PC-9821 (486+) |

**开发路径**：`/usr/local/bin/sdlnp21kai_sdl2`，配置 `~/.config/sdlnp21kai/np21kai.cfg`

### 启动链

```
NP2kai 内置 IPL ROM
    │ 从 HDD 读取 LBA 0
    ▼
IPL1 (LBA 0) — 魔数 "IPL1" @ offset 4:8
    │ 加载 IPL2 (LBA 1-13)
    ▼
IPL2 — 解析 LBA 1 的分区表 → 加载 VBR
    ▼
VBR (PC-98 格式，无需 55AA) → 加载 IO.SYS
    ▼
IO.SYS → MSDOS.SYS → CONFIG.SYS → AUTOEXEC.BAT → ENGINE.EXE
```

**关键细节**：
- PC-98 VBR 不需要 `0x55AA` 签名
- CONFIG.SYS 路径必须用 `\` 根相对格式（引导阶段盘符未分配）
- `msdos5.hdi` 的 FS：FAT16, 扇区 1024B, 每簇 2 扇区

## 3. engine.exe 构建与链接

### 辅助入口 `core/engine/test_hang.c`

最小退出测试用例：`int main(void) { return 0x42; }`。用于验证 MZ 头、链接脚本和 DOS 返回码通道正常。
正式项目使用 `projects/<name>/` 中的全功能入口，`test_hang.c` 仅用于最小冒烟测试。

### 工具链

| 工具 | 路径 | 用途 |
|------|------|------|
| `ia16-elf-gcc` | `/opt/gcc-ia16/bin/ia16-elf-gcc` | 编译 C → ELF |
| `ia16-elf-as` | `/opt/gcc-ia16/bin/ia16-elf-as` | 编译汇编 → ELF |
| `ia16-elf-ld` | `/opt/gcc-ia16/bin/ia16-elf-ld` | 链接 ELF → MS-DOS MZ |

### 链接器脚本 `core/msdos.ld`

```ld
OUTPUT_FORMAT("msdos")
OUTPUT_ARCH(i386)
SECTIONS
{
  . = 0x0;
  .text :
  {
    CREATE_OBJECT_SYMBOLS
    *crt0.o(.text)      /* 强制 _start 在 .text+0 */
    *(.text)
    *(.text.*)          /* 捕获 .text.startup（main） */
    etext = .;
  }
  .data : { *(.rodata) *(.rodata.*) *(.data) CONSTRUCTORS }
  .bss :
  {
    _bss_start = .;
    *(.bss) *(COMMON)
    _bss_end = ALIGN(4);
  }
  . = _bss_end + 0x800; /* 2KB stack */
  _stack_top = .;
  _end = .;
}
```

### 入口代码 `core/crt0.s`

```asm
.code16                  ; 必须加，默认是 32 位模式
.globl _start
_start:
    mov %ss, %ax         ; 复制 SS → DS, ES (小模式：所有段相同)
    mov %ax, %ds
    mov %ax, %es

    ; 清零 BSS
    movw $_bss_start, %di
    movw $_bss_end, %cx
    subw %di, %cx
    xorw %ax, %ax
    cld
    rep stosb

    ; 设置栈指针（SS 已 = DS，栈在 _end 后 2KB 空间）
    movw $_stack_top, %sp

    call main
    movb $0x4C, %ah      ; 退出
    int $0x21
```

## 4. 关键修复汇总

### 4.1 MZ 入口点修复

**问题**：`engine.exe` 的 MZ 头 `CS=0, IP=0` 指向 `.text+0`，但 `_start` 实际在偏移 `0x962`（因为 `crt0.o` 排在链接命令最后）。CPU 从 `.text+0` 的垃圾字节开始执行，触发 UD 异常。

**根因**：
1. 链接命令行 `crt0.o` 排在其他 `.o` 之后
2. 默认 `ld` 脚本只写 `*(.text)`，漏掉了 `*(.text.*)`，`-Os` 生成的 `.text.startup`（含 `main`）被丢弃，`call main` 留为未解析占位

**修复**：
- `msdos.ld`：`*crt0.o(.text)` 强制 `_start` 在 `.text+0`；`*(.text.*)` 捕获 `main`
- `crt0.s`：加 `.code16`（之前生成 `66 8c d0` 32 位前缀，错误）

### 4.2 栈与 BSS 修复

**问题**：BSS 变量（`buf`、`display_region_start`、`vsynced` 共 6 字节）未被清零；栈在镜像末尾，向下增长覆盖代码。

**修复**：
- `msdos.ld`：分三段（`.text` / `.data` / `.bss`），在 BSS 后 + `0x800`（2KB）栈空间，导出 `_stack_top`
- `crt0.s`：`rep stosb` 清零 BSS；`movw $_stack_top, %sp` 设栈指针

### 4.3 全屏填充修复

**问题**：`hal_video_fill_rect` 每平面只填 `8000 words`（640×200），但显示模式为 640×400，下半屏全黑。

**修复**：`core/plat/pc98/hal_pc98.c:93-96` — `8000` → `16000`

```
8000 words = 640 × 200 pixels  (旧，只能填上半)
16000 words = 640 × 400 pixels (新，全覆盖)
```

### 4.4 日志系统改进

**改动文件**：`core/engine/log.c` + `log.h`

| 改进 | 说明 |
|------|------|
| Buffer I/O | `log_buf[256]` 静态 buffer，满盘才 flush 到磁盘，IO 次数减 5-10 倍 |
| `log_printf` | 支持 `%s %d %x %c %%`，省去链式调用 |
| `log_nl` | `log_nl()` = `log_write("\r\n")` |
| `flush` | `log_flush()` 和 `log_close()` 自动 flush |

新增内存：~258 字节常驻（buffer + 指针） + 128 字节栈临时（printf）

## 5. 常见问题诊断

### 5.1 UD 异常（引擎启动即崩溃）

**症状**：NP2kai 启动后瞬间崩溃，日志有 `interrupt: num = 0x06`（UD 异常）。

**排查**：
```bash
# 检查 MZ 头
python3 -c "
import struct
with open('games/demo-A1/engine.exe','rb') as f:
    e=f.read()
cs=struct.unpack_from('<H',e,0x16)[0]
ip=struct.unpack_from('<H',e,0x14)[0]
ss=struct.unpack_from('<H',e,0x0E)[0]
sp=struct.unpack_from('<H',e,0x10)[0]
print(f'CS={cs} IP={ip} SS={ss} SP={sp}')
img_start=struct.unpack_from('<H',e,0x08)[0]*16
print(f'Entry bytes: {\" \".join(f\"{b:02x}\" for b in e[img_start:img_start+16])}')
"

# 反汇编
ia16-elf-objdump -d -m i8086 games/demo-A1/engine.exe | head
```

**常见原因**：
1. `_start` 不在 `.text+0` → 检查 `*crt0.o(.text)` 是否在链接脚本最前
2. `call main` 未解析 → 检查 `*(.text.*)` 是否包含 `.text.startup`
3. 错误的 CPU 模式 → 检查 crt0.s 是否有 `.code16`

### 5.2 栈溢出

**症状**：运行正常但某次调用后随机崩溃；或 BSS 变量值异常被改写。

**排查**：检查 MZ 头 `SS` 和 `SP`：
- `SS=0`（同段）是正常的
- `SP = _stack_top`（BSS + 2KB）确保栈在镜像之后的安全区域

### 5.3 画面显示异常

**症状**：半屏渲染、颜色错误、画面错位。

**排查**：
1. 确认 GDC 显示模式（`hal_video_init` 中 `640×400`）
2. 确认 `fill_plane` 的 word 计数：
   - 640×200 = `8000 words/plane`
   - 640×400 = `16000 words/plane`
   - 640×480 = `19200 words/plane`

### 5.4 日志看不到

**症状**：引擎跑了但 `ENGINE.LOG` 为空或没有生成。

**排查**：
1. 是否调用了 `log_open()` → 文件可能未创建
2. `log_write()` 是否调用 → buffer 可能未 flush（调 `log_close()` 或 `log_flush()`）
3. 是否在返回前调了 `log_close()`
4. HDI 是否用最新 engine.exe 重建

## 6. HDI 制作参考

### 基座镜像：`tools/msdos5.hdi`

- MS-DOS 5.0 48MB HDI
- FAT16, 扇区 1024B, 每簇 2 扇区
- 保留 IPL/VBR 完整启动链

### 注入流程

```
tools/ref_disk/    →  系统文件 (IO.SYS, MSDOS.SYS, COMMAND.COM, DOS/*)
tools/ref_config/  →  CONFIG.SYS, AUTOEXEC.BAT
games/<name>/      →  游戏文件 (engine.exe, 素材...)
        │
        ▼
NAIZFatFS.write_back_from_directory()
    │ 全部写入 FAT 文件系统
    │ IO.SYS 强制排根目录第一
    │ 写回 HDI 容器
    ▼
disks/<name>.hdi
```

### 关键约束

| 约束 | 说明 |
|------|------|
| CONFIG.SYS 路径 | 必须用 `\` 根相对（无盘符），引导阶段盘符尚未分配 |
| IO.SYS 根目录第一 | `write_back_from_directory()` 强制排序，否则 MS-DOS 无法启动 |
| 8.3 文件名 | FAT 文件系统要求，引擎文件必须符合 |
